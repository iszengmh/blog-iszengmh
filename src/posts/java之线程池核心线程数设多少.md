## 核心线程数公式和对比

大部分的web项目只有CURD和http请求,所以基本都是IO密集型

||CPU 密集型|IO 密集型|
|--|--|--|
|公式|线程数 = CPU 核心数 + 1（加 1 是为了防止某个线程阻塞时 CPU 不闲置）|线程数 = CPU 核心数 × 2（IO 等待越长 倍数越大 不是死套 2n！）|
|精确公式|N/A|线程数 = CPU 核心数 × (1 + 等待时间 / 计算时间)|
|优点|纯计算任务 CPU 拉满|CPU 空闲，多开线程利用率更高|
|场景|纯内存计算 无外部调用，CPU使用率接近100%，加线程只会更慢 徒增上下文切换|网络请求、调数据库、Redis、第三方接口，线程大部分在WAITING 加线程能提吞吐 但有上限|
|如何判断|1. 用Arthas jprofiler看线程状态分析是CPU密集型，还是IO密集型; 2. 压测看CPU占用|


## 线程数设多了 反而更慢
很多人以为线程越多并发越高 大错特错！
最大坑：上下文切换
线程数超过CPU核心数
CPU要不停保存恢复线程状态
线程数 = CPU核心数：切换几乎为0
线程数 = 2×CPU核心数：切换明显
线程数 = 4×CPU核心数：
CPU30% 性能都浪费在切换上！
盲目加大线程数 OOM 响应慢 CPU飙高全来了！

## 队列 & 拒绝策略

### 队列对比
|队列|描述|缺点|
|--|--|--|
|LinkedBlockingQueue：|无界 不触发拒绝|极高风险导致 OOM、上游请求激增时，只能被动等待内存耗尽或服务崩溃|
|ArrayBlockingQueue：|有界 防OOM 有背压 需合理设大小|需合理设大小，太小会导致不必要的阻塞或拒绝，太大则浪费内存|
|SynchronousQueue：|不缓存 直接交线程 拒绝策略|易触发拒绝 |

注：
LinkedBlockingQueue：即可以是有界，也可以是无界（当容量设置为Integer.MAX_VALUE时）
有界：队列有固定的最大容量，创建时必须指定容量上限
无界：逻辑上不限制容量，可以一直添加元素直到内存耗尽。
背压（backpressure）：生产者速度会被下游慢消费自然限制住，防止系统过载。

|策略|描述|缺点|
|--|--|--|
|AbortPolicy：|抛异常 关键业务用|N/A|

### 队列的业务场景

一、队列的业务场景
1. LinkedBlockingQueue（无界）
场景：任务量可预估、积压可控、允许暂时等待但绝对不能拒绝的业务。

例子：
某财务系统的对账文件异步处理。上游系统每天固定时段推送几百个对账文件，处理速度稳定。
使用 LinkedBlockingQueue（无界）接收文件路径，后台线程池逐一处理。由于任务总量有限且内存足够，不会 OOM。
注意：生产环境慎用无界队列，必须有监控和限流措施。

2. ArrayBlockingQueue（有界）
场景：资源敏感、需要背压保护、防止下游被打垮的业务。

例子：
一个短信发送服务，业务高峰期每秒可能请求 10000 条短信，但下游短信通道每秒只能处理 2000 条。
使用 ArrayBlockingQueue（容量 5000）作为缓冲，队列满时生产者阻塞等待，自然形成背压，避免通道过载。
合理设大小：队列太短会频繁阻塞影响吞吐，太长浪费内存。

3. SynchronousQueue（无缓存，直接移交）
场景：任务必须立即被处理，不允许排队等待，否则直接拒绝并让上游重试或降级。

例子：
一个实时风控决策引擎：每笔支付请求需要立即同步调用风控规则（超时 <50ms）。
使用 SynchronousQueue 配合固定大小线程池，如果所有 worker 线程都在忙，新请求直接触发拒绝策略（如 AbortPolicy），上游服务捕获异常后走兜底逻辑（例如直接放行或转人工审核）。
这样保证了请求的实时性，绝不积压。

### 拒绝策略的业务场景（配合有界队列或 SynchronousQueue）
1. AbortPolicy（抛异常）
场景：关键业务，不能无感知地丢失任务，必须显式处理拒绝情况。

例子：
银行转账服务：队列满时抛出 RejectedExecutionException，调用方捕获后记录失败事件、回滚事务、发送告警，最终人工介入或重试。
绝不允许静默丢弃一笔转账。

2. CallerRunsPolicy（调用者执行）
场景：削峰填谷，放缓任务提交速度，保证系统不崩溃。

例子：
一个网关限流降级场景：当请求队列满时，新任务由调用者线程（例如 Tomcat 工作线程）自己执行。
这样做会拖慢请求线程，进而减少新请求的接收速率，达到保护后端的效果。常用于非关键但需要保活的业务。

3. DiscardPolicy（静默丢弃）
场景：可容忍少量数据丢失、高频且价值较低的业务。

例子：
用户行为埋点日志：每秒产生数万条点击事件，允许丢失千分之一。
当队列满时直接丢弃新事件，确保监控系统的稳定性，不影响主要业务。

4. DiscardOldestPolicy（丢弃最老）
场景：高实时性要求，允许丢弃旧数据来换取最新数据的处理机会。

例子：
实时股票行情推送：如果队列满了，丢弃队列中最早的一条行情数据，然后将最新行情放入队列。
因为对于行情来说，最新的价格比过时的价格更有价值，旧行情被丢弃是可以接受的。
